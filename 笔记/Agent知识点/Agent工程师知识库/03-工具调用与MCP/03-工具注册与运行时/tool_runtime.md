面试官，关于 tool_runtime，我从本质定义、核心职责、架构分层、关键工程机制以及与 ToolRegistry 的关系这五个层面来说明。

**一、本质定义**

tool_runtime 是工具调用的**执行环境与策略引擎**。它接收来自 ToolRegistry 的调用请求（工具名 + 已校验的参数），负责在受控的运行时中真正执行该工具的逻辑，并返回结果。与 ToolRegistry 的区别在于：Registry 管的是“这个工具是什么、参数长什么样、能不能调用”；tool_runtime 管的是“怎么执行、在哪里执行、执行失败了怎么办”。两者是“元数据”与“运行时”的分工。

**二、核心职责**

tool_runtime 承担四方面职责。第一，**执行隔离**——为每个工具调用提供独立的运行边界，防止工具之间的资源冲突和副作用污染。第二，**策略执行**——统一实现超时控制、重试、熔断、限流、并发控制等非功能性策略，避免每个工具自己重复实现。第三，**安全性保障**——限制工具的权限（文件系统、网络、环境变量），防止恶意或错误的工具调用破坏宿主系统。第四，**可观测性**——记录每次调用的输入、输出、耗时、错误栈，供审计和调试。

**三、架构分层**

典型的 tool_runtime 采用分层执行模型。最上层是**调度层**，接收调用请求，根据工具类型（同步/异步、本地/远程）选择合适的执行器。第二层是**策略层**，以装饰器或管道的方式依次应用超时、重试、熔断、限流等策略。第三层是**沙箱层**，负责资源隔离——对于本地函数，可能限制 CPU 时间、内存；对于子进程或脚本，可能丢入容器或受限的用户命名空间；对于远程 HTTP 调用，则控制网络出站规则。最底层是**适配器层**，真正执行工具实现：调用本地函数、发起 HTTP 请求、执行 SQL 查询、或通过 MCP 客户端调用远端服务。

**四、关键工程机制**

这里有几个值得深入的点。

第一，**执行模式的差异化**。工具分为同步和异步两类。同步工具在调用结束前阻塞当前协程或线程；异步工具立即返回一个 task_id，由 runtime 后台执行，Agent 需要轮询或通过回调获取结果。tool_runtime 需要透明地支持这两种模式，对上层统一返回一个 Awaitable 或 Future。

第二，**策略链的可组合性**。tool_runtime 通常实现为中间件管道：请求先经过限流中间件（判断是否超过 QPS 限制），再经过重试中间件（根据错误类型决定是否重试），最后进入超时中间件（设置整体执行 deadline）。这种设计使得策略可以灵活增删，且与工具的业务逻辑解耦。

第三，**沙箱的层次化隔离**。在多租户或高安全场景下，tool_runtime 不能只靠进程内隔离。常见的做法是：对于高风险工具（执行 shell 命令、写文件），通过子进程 + seccomp / AppArmor 限制系统调用；对于更高级别的风险，甚至将工具调度到独立的容器或轻量级 VM 中执行。tool_runtime 负责根据工具的“风险等级”标签动态选择沙箱级别。

第四，**资源回收与超时传播**。工具可能泄漏文件句柄、未关闭的连接或子进程。tool_runtime 必须为每次调用设置一个“最终回收”机制——无论成功或失败，都要释放临时资源。超时控制需要从调用入口一直传播到底层适配器，防止某个环节阻塞导致整个调用挂死。

**五、与 ToolRegistry 的关系**

在实际架构中，tool_runtime 和 ToolRegistry 通常是紧耦合的。调用链为：Agent 发起 Tool Calling → ToolRegistry 完成参数校验和权限检查（whitelist）→ 得到“工具句柄”（包含工具的实现引用和执行元数据）→ 将句柄连同参数一起交给 tool_runtime 执行。Registry 负责“能不能调”，runtime 负责“怎么调得好”。两者共同构成了 Tool Calling 执行引擎的后半段。

**总结**

tool_runtime 的本质可以概括为：它是工具执行的**统一控制面**，将安全性、可靠性、可观测性从工具业务逻辑中剥离出来，作为横切关注点集中实现。没有它，每个工具都需要自己处理超时、重试、沙箱和日志，系统将变得混乱且脆弱。

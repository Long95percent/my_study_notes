

## 问题 1：项目总架构与个人边界

  

你简历里写的是：

  

> Jarvis Life 多智能体系统，技术栈 Python FastAPI / asyncio / LangGraph，Java Gateway，SQLite，SSE。

  

请你先用 2 到 3 分钟讲清楚这个系统的整体架构，要求包括：

  

1. 用户从前端发起一次请求后，请求链路是怎么走的？

2. 你个人主要负责哪些模块？

3. 你说系统从“对话型”升级为“可执行型 Agent”，这个升级前后最大的架构差异是什么？

  

### 你的回答

  

这个项目可以分成三层：前端交互层、Java Gateway 聚合层，以及 Python AI Agent 执行层。

  

用户在前端发起一次对话或任务请求后，前端会把用户输入、会话 ID、用户 ID、是否需要流式输出等信息发到 Java Gateway[[JAVA后端#^javagateway]]。Gateway 主要做统一入口和协议适配，比如参数规范化、请求聚合、SSE 转发，以及把前端和 Python AI 服务解耦。之后 Gateway 会把请求转发给 Python FastAPI 服务。

  

进入 Python 服务后，请求先进入 intent_router[[智能体系统架构#^1]]。intent_router 会结合用户原始输入、最近会话上下文、可用 Agent 列表，判断这次请求应该由哪个主 Agent 处理，是否需要 consultation。确定主 Agent 后，请求进入对应 Agent 的 LangGraph 执行链路。Agent 如果需要查日程、写日程、写记忆或产生主动消息，不会直接执行任意函数，而是通过 ToolRegistry 找到工具定义，再经过 tool_runtime 做参数校验、白名单权限校验、异常包装和结果标准化，最后才真正访问 SQLite、记忆模块或日程模块。

  

返回链路上，如果是普通问答，Python 会返回结构化 JSON 给 Gateway，再返回前端；如果是流式回答，就通过 SSE 把 token 或阶段性事件逐步推回前端。比如 Maxwell 写日程这种场景，后端不只是返回一段文本，还会返回待确认卡片、日程写入结果，前端 CalendarPanel 再刷新展示。

  

我个人主要负责的是 Python AI 侧的 Agent 执行架构，包括 ToolRegistry、tool_runtime、whitelist、intent_router，以及 Maxwell 日程秘书链路、长期记忆和主动消息持久化这几块。Java Gateway 我主要参与的是统一调用链路联调、接口契约、SSE 转发和稳定性治理

  

我说从“对话型”升级到“可执行型 Agent”，核心差异是系统具备了可控地改变外部状态的能力。以前系统主要是根据用户输入生成自然语言回复，回答完就结束；升级之后，Agent 可以在权限受控的前提下调用工具，写入日程、生成待确认卡片、更新长期记忆、产生主动消息。为了让这个过程可控，我加了工具注册、白名单隔离、参数校验、异常处理、确认机制和回归测试。这样系统从“会聊天”变成了“能在安全边界内完成任务”。

  

### 参考回答要点

  

- 完整链路：Frontend -> Java Gateway -> Python FastAPI -> intent_router -> Agent / LangGraph -> ToolRegistry / tool_runtime -> SQLite / memory / schedule -> SSE / JSON。

- 个人边界要具体，不要把协作部分说成独立完成。

- “可执行型 Agent”要强调工具、权限、确认、状态变化和测试闭环。

## 问题 2：ToolRegistry、tool_runtime、whitelist 的设计

  

你写到：

  

> 构建 ToolRegistry + tool_runtime + whitelist + intent_router，实现 6 个 Agent 的 Tool Calling 与权限隔离。

  

请你具体讲一下这几个模块之间的关系，并按 Maxwell 写日程的链路说明。

  

### 你的回答

  

我当时把这几个模块分成三层来看：ToolRegistry 是工具元信息中心，tool_runtime 是工具执行层，whitelist 是权限边界层。intent_router 负责决定请求先交给哪个 Agent，而 Agent 真正要执行外部动作时，就必须经过 ToolRegistry、whitelist 和 tool_runtime。

  

ToolRegistry 里注册的是一整套工具定义。通常包括工具名、工具描述、参数 schema、返回格式、实际 handler、工具所属领域，以及权限标签。工具 schema 一方面给 LLM 看，让模型知道可以调用什么工具、需要什么参数；另一方面也给运行时校验用，避免模型生成缺字段、错字段或者类型不匹配的调用。

  

tool_runtime 负责真正执行工具。它会先根据工具名到 ToolRegistry 查工具是否存在，然后根据当前 Agent、用户上下文和工具白名单判断是否允许调用。通过权限检查后，再根据 schema 做参数校验和必要的默认值补全。之后 runtime 会统一处理同步和异步工具：如果 handler 是 async，就 await；如果是普通同步函数，就按统一接口包装。工具执行过程中如果发生异常，runtime 不会把底层异常直接暴露给 Agent，而是包装成标准结果，比如 success、error_code、message、retryable、data。这样 Agent 可以根据标准结果决定重试、反问用户，还是降级回复。

  

whitelist 主要解决权限隔离。最基础的是按 Agent 维度配置可调用工具，比如 Maxwell 可以调用日程查询、日程创建、待确认卡片工具，但不能调用健康 Agent 专属的写入工具，也不能随便改长期记忆中不属于它边界的数据。如果某个 Agent 试图调用不在白名单里的工具，runtime 会直接拒绝，记录日志，并返回一个权限错误，而不是继续让模型尝试执行。

  

我没有直接让 LangGraph 或 LLM 自己决定工具调用，是因为 LLM 有几个风险：第一，它可能幻觉出不存在的工具名；第二，它可能生成不合法参数；第三，它可能越权调用其他 Agent 的工具；第四，一些工具有副作用，比如写日程、写记忆，不能完全依赖模型自觉。所以我把“模型想调用什么”和“系统允许调用什么”分开，模型只提出意图，真正执行必须过 runtime 和 whitelist。

  

以 Maxwell 写入日程为例，用户说“明天下午三点提醒我参加组会”。请求先被 intent_router 路由到 Maxwell。Maxwell 识别这是创建日程意图，然后生成类似 create_calendar_event 的工具调用，参数包括 title、start_time、duration、source_text 等。tool_runtime 收到调用后先查 ToolRegistry，确认工具存在；再检查 Maxwell 是否在白名单里有这个工具权限；然后校验 start_time 是否能解析、title 是否为空、时间是否合理。如果信息完整且是低风险操作，可以生成待确认卡片；如果缺少关键信息，比如没有具体日期，就反问用户。用户确认后，runtime 再调用真正写 SQLite 的 handler，写入成功后返回结构化结果，Gateway 再通知前端 CalendarPanel 刷新。整个过程中，LLM 没有直接写库，真正的副作用都在 runtime 控制下发生。

  

### 参考回答要点

  

- ToolRegistry 是工具元信息中心。

- tool_runtime 是执行层，负责查找、校验、鉴权、执行、异常包装。

- whitelist 是安全边界，避免 Agent 越权。

- 不能完全信任 LLM 的工具选择。

## 问题 3：intent_router 的路由准确性与失败处理

  

你简历里提到：

  

> 构建 ToolRegistry + tool_runtime + whitelist + intent_router，实现 6 个 Agent 的 Tool Calling 与权限隔离。

> 多 Agent 协同机制 consultation 与角色主权回复策略：解决跨领域任务路由失真问题。

  

请你结合 6 个 Agent 说明 intent_router 的输入输出、实现方式、失败处理和评估。

  

### 你的回答

  

intent_router 的作用是把用户请求分配给最合适的主 Agent，并判断是否需要多 Agent consultation。它不是简单返回一个 Agent 名字，而是输出一个结构化路由结果。

  

输入一般包括用户原始消息、最近几轮会话上下文、用户当前状态、可用 Agent 列表、部分长期偏好，以及上一轮是否已经有未完成任务。输出我会设计成 target_agent、intent、confidence、reason、candidate_agents、need_consultation 这样的结构。这样后面不是只能盲目相信一个分类结果，而是可以根据置信度和候选 Agent 做后续处理。

  

实现上我采用的是规则和 LLM 分类混合的思路。像“帮我添加日程”“明天提醒我”这种高确定性的日程意图，可以用关键词和规则先命中 Maxwell；像“帮我安排下周复习 Java 面试，同时别和健身计划冲突”这种跨领域任务，就不适合只用规则，因为它既涉及学习规划，也涉及日程，也涉及健康或运动约束，这种会交给 LLM 分类，并让它输出主 Agent 和候选 Agent。规则的优点是稳定、便宜、可回归；LLM 的优点是能处理复杂语义，所以我没有只选一种。

  

我说的“跨领域任务路由失真”，典型例子就是用户说：“帮我安排下周复习 Java 面试，同时别和健身计划冲突。”如果 router 只看到“复习 Java 面试”，可能会路由到学习 Agent；只看到“安排下周”，可能路由到 Maxwell；只看到“健身计划”，可能路由到健康 Agent。实际上这个请求应该由 Maxwell 或学习规划 Agent 作为主 Agent，然后咨询健康 Agent 获取健身约束，最后由主 Agent 统一生成日程计划。也就是说，不是每个 Agent 都直接对用户说话，而是主 Agent 负责最终回复和工具调用。

  

如果 router 置信度高，就直接路由到目标 Agent。如果置信度中等，或者 candidate_agents 里有多个强相关 Agent，就触发 consultation。如果置信度很低，系统会优先反问用户，比如“你是想创建日程，还是只是让我给一个学习建议？”而不是强行执行工具。对于涉及写日程、写记忆这种有副作用的动作，即使路由到了目标 Agent，也需要后续确认机制兜底。

  

评估方面，我不会只看几个 demo 是否跑通。我会积累典型用户请求作为路由回归用例，比如纯日程、纯问答、学习计划、健康建议、跨领域任务、模糊任务等。每次改 router prompt 或规则后跑回归，看 target_agent、need_consultation、intent 是否符合预期。同时也会从日志里抽样错路由 case，记录用户原话、错误路由结果、期望路由结果，再补充到测试集里。这样 router 是可迭代的，而不是凭感觉调 prompt。

  

### 参考回答要点

  

- 输出最好是结构化路由结果，不只是 Agent 名称。

- 规则 + LLM 分类混合更工程化。

- 低置信度要 consultation、反问或降级。

- 评估要有回归用例和错路由复盘。

## 问题 4：consultation 机制与角色主权回复策略

  

你简历里写到：

  

> 多 Agent 协同机制（consultation）与角色主权回复策略：解决跨领域任务路由失真问题，在协作效率与安全边界之间实现工程化平衡。

  

请你说明 consultation 什么时候触发、怎么调用、谁最终回复、如何控制成本。

  

### 你的回答

  

consultation 的目的不是让所有 Agent 都回答一遍，而是在跨领域任务里让主 Agent 获得其他 Agent 的专业约束。它主要在三种情况下触发：第一，router 置信度不高，但有多个候选 Agent；第二，用户请求天然跨领域，比如学习计划不能和健身计划冲突；第三，主 Agent 执行过程中发现自己缺少其他领域信息，比如 Maxwell 排日程时需要健康 Agent 给出运动时间限制。

  

调用方式上，我更倾向于并行咨询少量相关 Agent，而不是串行链式调用所有 Agent。router 会给出 candidate_agents，然后系统根据任务类型最多选两到三个相关 Agent。被咨询的 Agent 通常进入轻量 consultation 模式，只返回结构化建议，比如 constraints、risks、suggestions、confidence，而不是完整执行自己的工具链。这样可以控制成本和延迟。

  

权限上，被咨询 Agent 默认是只读或建议模式，不允许随便产生有副作用的工具调用。比如健康 Agent 可以告诉 Maxwell：“用户通常晚上 7 点到 8 点健身，这段时间不建议安排学习任务”，但它不能直接修改日程，也不能直接写入新的健康计划。真正写日程、创建待确认卡片、更新任务状态的动作，应该由主权 Agent 在用户确认后执行。

  

“角色主权回复策略”指的是：每次对外只应该有一个 primary / sovereign Agent 对最终回复负责。其他 Agent 只是提供建议，不能抢答。比如用户问“帮我安排下周复习 Java 面试，同时别和健身计划冲突”，系统可以选择 Maxwell 作为主权 Agent，因为最终要落到日程安排。学习 Agent 提供复习内容建议，健康 Agent 提供健身时间约束，最后由 Maxwell 合并这些建议，生成一个可确认的日程方案。

  

如果不同 Agent 的建议冲突，我会按优先级处理：用户当轮明确指令最高，其次是安全和已有日程硬约束，再其次是长期偏好，最后才是 Agent 建议。比如学习 Agent 建议晚上 7 点复习，但健康 Agent 提醒用户 7 点到 8 点固定健身，那么 Maxwell 应该避开这个时间，而不是直接听学习 Agent。

  

为了避免 consultation 成本和延迟失控，我会做几件事：限制参与 Agent 数量，比如最多 2 到 3 个；给每个 consultation 设置超时；让被咨询 Agent 返回结构化短结果，而不是长文本；对于低价值或高确定性任务不触发 consultation；必要时缓存短期约束，比如当天已经查过的健身计划，不用重复问。这样 consultation 是一个受控机制，不是“多 Agent 越多越好”。

  

用刚才的例子，“帮我安排下周复习 Java 面试，同时别和健身计划冲突”，router 识别为跨领域任务，选择 Maxwell 作为主权 Agent，候选咨询 Agent 是学习 Agent 和健康 Agent。学习 Agent 返回建议：Java 基础、Spring、项目复盘、八股训练分成几个模块。健康 Agent 返回约束：周一三五晚上 7 点到 8 点健身，最好不要安排高强度学习。Maxwell 合并后生成下周学习日程，避开健身时间，最后生成待确认卡片。用户确认后，Maxwell 才调用日程写入工具。

  

### 参考回答要点

  

- consultation 要有触发条件和数量限制。

- 被咨询 Agent 默认只读。

- 主权 Agent 负责最终回复和副作用工具调用。

- 冲突处理要有优先级。

## 问题 5：Maxwell 日程秘书 Agent 的端到端闭环

  

你简历里写到：

  

> Maxwell 日程秘书 Agent 链路：从自然语言意图识别、上下文补全、工具调用，到长期计划拆解、日程写入、待确认卡片、前端 CalendarPanel 展示闭环。

  

请你用具体例子完整说明。

  

### 你的回答

  

Maxwell 不是简单的“创建日程工具”，而是一条从意图识别到前端展示的完整闭环。我把它拆成识别层、补全层、计划层、确认层、执行层和展示层。

  

首先是识别层。用户请求进入后，intent_router 会先判断是否属于日程领域，如果是，就交给 Maxwell。Maxwell 内部还会做二级意图识别，区分是创建日程、查询日程、修改日程、删除日程、长期计划拆解，还是普通建议。二级识别结果最好是结构化的，比如 action=create_event 或 plan_split，confidence、required_fields、missing_fields、risk_level 等。

  

第二是上下文补全。日程任务最关键的是时间信息。像“明天下午”要结合用户时区和当前日期解析；“下周一晚上”要解析成具体日期和时间段；“每天早上”要转换成重复规则。对于缺失字段，我会分成可默认和必须反问两类。比如地点可以为空，优先级可以默认 normal，持续时间可以根据任务类型给默认值；但开始时间完全缺失、重复规则不清楚、或者用户说“找个合适时间”但已有日程冲突严重，就需要反问或生成可选方案。

  

第三是长期计划拆解。比如用户说“未来两周帮我准备 Java 面试”，Maxwell 不应该直接写一个大日程，而是先把目标拆成多个学习块，比如 Java 基础、集合并发、Spring Boot、项目复盘、模拟面试。拆解时要考虑已有日程、用户偏好和任务优先级。比如用户偏好晚上学习，就优先排晚上；如果某些晚上已有健身计划，就避开。拆出来的计划一般不会直接写库，而是先生成待确认卡片，让用户确认后再执行。

  

第四是待确认卡片。因为写日程是有副作用操作，我会用确认卡片隔离风险。卡片状态可以设计成 pending、confirmed、cancelled、expired。卡片里保存计划内容、原始请求、生成时间、过期时间和 request_id。用户确认后，后端根据 card_id 和 request_id 做幂等校验，避免用户重复点击导致重复写入。如果用户修改卡片内容后再确认，后端会把修改后的结构化内容重新校验一遍，再写入日程。

  

第五是执行和展示。确认后，Maxwell 通过 tool_runtime 调用日程写入工具，写入 SQLite。写入成功后返回结构化结果，比如 created_event_ids、failed_items、summary。前端 CalendarPanel 可以主动重新拉取日程，也可以通过 SSE 或主动消息收到刷新事件。如果写入成功但前端刷新失败，后端至少要返回明确的写入成功结果，前端可以提示用户刷新，或者下次进入 CalendarPanel 时重新拉取。

  

如果出现部分成功部分失败，我会根据任务类型选择策略。对于一批强相关日程，比如同一个长期计划拆出来的 10 个日程，如果要求强一致，可以用事务整体提交，失败就回滚；如果允许部分成功，就必须告诉用户哪 7 个成功、哪 3 个失败，并支持重试或重新生成失败项。不能只说“安排好了”，否则用户会以为全部成功。

  

举个例子，用户说：“下周帮我每天晚上 8 点安排 1 小时 Java 面试复习，但周三不要安排。”router 先路由到 Maxwell。Maxwell 二级识别为长期计划拆解 + 批量创建日程。补全层解析“下周”“每天晚上 8 点”“1 小时”“周三排除”，生成周一、周二、周四、周五、周六、周日的候选时间块。然后检查已有日程冲突，如果某天晚上 8 点已有安排，就标记冲突并给替代时间。接着生成待确认卡片，把 6 个日程列出来。用户确认后，tool_runtime 校验 Maxwell 有写日程权限，再批量写 SQLite。写入成功后返回 event_ids，并通知前端 CalendarPanel 刷新。这样才算真正形成了从自然语言到日程展示的闭环。

  

### 参考回答要点

  

- Maxwell 要讲完整闭环，而不是单次工具调用。

- 时间解析、确认卡片、幂等性、冲突处理是重点。

- 部分失败要有事务或补偿策略。

## 问题 6：长期记忆链路、记忆边界与持续学习

  

你简历里写到：

  

> 构建长期记忆与偏好学习链路：边界化召回 + 后台异步抽取/压缩，并将主动消息从内存队列升级为持久化机制，提升多端一致性与系统稳定性。

  

请你讲清楚系统记住什么、怎么写入、怎么召回、怎么持续学习。

  

### 你的回答

  

我这里说的长期记忆不是简单把所有聊天记录拼进 prompt，而是把记忆分层管理。主要有几类：第一类是用户偏好，比如作息、学习时间偏好、饮食习惯；第二类是事实记忆，比如用户的长期目标、重要日期、固定计划；第三类是任务记忆，比如未完成事项、待确认卡片、主动消息；第四类是会话摘要或 Agent 运行轨迹，用来帮助后续理解上下文。这些类型在存储时应该有 memory_type 区分，而不是都存成一段 summary。

  

写入链路上，我不会每轮对话都同步写长期记忆。主对话链路应该优先保证响应速度，所以我采用后台异步抽取和压缩。具体来说，用户一轮对话结束后，系统先判断这轮内容是否有记忆价值，比如是否包含稳定偏好、长期目标、明确任务状态，或者用户对系统建议的反馈。如果有，就把原始对话片段放入后台任务。后台任务调用抽取逻辑，把候选记忆提取成结构化格式，比如 content、memory_type、source、confidence、created_at、expires_at。然后做去重、合并和压缩，再写入持久化存储。如果异步任务失败，至少要记录失败状态，后续可以重试，不能影响主对话返回。

  

“边界化召回”指的是记忆召回不能无边界地把所有用户历史都拿出来，而是要按用户、Agent、任务域、权限和时间有效性过滤。比如 Maxwell 日程 Agent 默认只召回和日程、任务、学习时间偏好相关的记忆，不应该随便召回健康 Agent 的敏感记录。但如果用户请求明确涉及“不要和健身计划冲突”，那么 Maxwell 可以在权限允许的范围内召回健康计划的时间约束，比如健身时间段，而不是召回全部健康细节。这样既能协作，又能避免记忆污染和隐私越界。

  

记忆冲突和过期也很重要。比如用户以前说“我晚上 11 点睡”，后来又说“最近改成 12 点睡”，系统不能永远相信第一条。我的处理方式是给记忆加时间戳、置信度和状态。新的明确偏好可以覆盖旧偏好，或者把旧偏好标记为 inactive。对于“最近”这类短期偏好，可以设置 expires_at 或较低长期置信度。召回时结合新鲜度和置信度排序。这样记忆不是永久不变的，而是可以更新、衰减和失效。

  

偏好学习的闭环是：观察用户表达或行为，抽取候选偏好，存储为弱记忆，后续在合适任务中召回并影响决策，再根据用户反馈修正。如果用户多次说“我晚上效率更高，Java 面试复习尽量安排在晚饭后”，系统可以先抽取一条学习时间偏好：用户偏好晚饭后安排 Java 面试复习。第一次可以作为弱偏好，如果后续多次出现或者用户确认，就提升置信度。之后 Maxwell 排学习日程时召回这条偏好，优先安排晚饭后。如果用户后来拒绝说“最近晚上太累，改到早上”，系统就要更新偏好，而不是继续强化旧偏好。

  

主动消息从内存队列升级为持久化机制，解决的是可靠性和多端一致性问题。内存队列的问题是服务重启会丢消息，多端登录时状态不一致，也不好记录消息是否已读或已处理。持久化后，每条主动消息可以有状态，比如 pending、delivered、read、handled、expired、cancelled。服务重启后可以从数据库恢复未投递消息；用户手机端已读后，电脑端也能看到状态变化；过期提醒不会一直重复推送。这个改变让主动消息从临时内存事件变成了可追踪的任务状态。

  

用一个例子来说，用户连续几天说“我晚上效率更高，Java 面试复习尽量安排在晚饭后”。系统不会每次都简单拼进 prompt，而是在后台抽取出一条偏好：memory_type=preference，domain=study_schedule，content=用户偏好晚饭后安排 Java 面试复习，confidence 随着多次出现提高。之后用户让 Maxwell 安排下周复习计划时，Maxwell 召回这条偏好，同时查询已有日程和健身时间，优先在晚饭后没有冲突的时间段安排复习。如果用户接受，说明偏好有效；如果用户修改，系统再反向更新偏好。这就是我理解的持续学习闭环。

  

### 参考回答要点

  

- 长期记忆要分层，不是 conversation summary。

- 写入要异步抽取、去重、压缩。

- 召回要有用户、Agent、任务域、权限和时间边界。

- 持续学习是偏好闭环，不等于微调模型。

- 主动消息持久化要有状态机和多端一致性。

## 问题 7：RAG、记忆检索与效果评估

  

你简历里虽然没有直接写 RAG，但提到了长期记忆、边界化召回、偏好学习。请你说明长期记忆召回和 RAG 的关系，以及如果设计 RAG 记忆检索层会怎么做。

  

### 你的回答

  

我的理解是，长期记忆召回和普通 RAG 有相似点，都是在模型生成或 Agent 决策前，从外部存储中检索相关信息，再放入上下文。但它们的重点不一样。普通 RAG 更多面向文档知识库，比如公司文档、产品手册、论文资料；而 Jarvis Life 的记忆检索更像个人记忆库和任务状态库，里面有用户偏好、历史计划、日程、待办、主动消息、会话摘要等。它不仅影响模型怎么回答，还会影响 Agent 怎么规划、是否调用工具、什么时候发主动消息，所以错误召回的风险更高。

  

如果让我给 Jarvis Life 加一个更完整的 RAG 记忆检索层，我会按“结构化过滤 + 向量检索 + 重排”的方式设计。数据源包括历史对话摘要、用户偏好、长期目标、日程、任务状态、主动消息，以及用户上传或配置的文档。chunk 粒度不能太粗，也不能太碎。对于对话，可以按一轮或一个主题压缩成记忆条目；对于偏好，最好是一条偏好一条 memory；对于日程和任务，本身就是结构化对象，不一定要完全当文本 chunk，而是保留结构化字段。

  

metadata 很关键，不能只存 text 和 embedding。我会至少保存 user_id、agent_id 或 memory_scope、memory_type、domain、created_at、updated_at、expires_at、confidence、source、permission_scope、last_used_at。检索时第一步先做结构化过滤，比如必须 user_id 匹配，不能跨用户；根据当前 Agent 和任务域过滤 memory_scope；过滤掉 expired 的记忆；对于敏感记忆还要检查权限。第二步再在过滤后的候选里做向量召回。第三步结合相似度、新鲜度、置信度、任务相关性和用户确认程度做 rerank。

  

召回排序不能只看 embedding 相似度。比如用户问“帮我安排一周学习计划”，一条很老的“我喜欢晚上学习”可能语义相似，但如果用户最近说“这周晚上要加班”，那新的记忆应该优先。排序时我会综合：语义相关性、时间新鲜度、置信度、是否用户确认、是否和当前任务强相关、是否在当前 Agent 权限范围内。必要时可以用规则重排，比如当前用户明确指令永远高于历史记忆。

  

为了防止 RAG 或记忆召回带来错误增强，我会做几层保护。第一，召回前过滤过期和越权记忆。第二，召回后如果历史记忆和用户当前明确指令冲突，以当前指令为准。第三，对于低置信度记忆，只作为参考，不直接驱动高副作用工具。第四，对于可能影响日程写入、长期计划的记忆，最好在待确认卡片里可解释地展示出来，比如“我根据你偏好晚饭后学习，为你安排了晚上 8 点”。如果用户不认可，就可以纠正记忆。

  

评估方面，不能只看 recall@k。离线可以构造测试集，比如给定用户问题和记忆库，标注哪些记忆应该被召回，计算 recall@k、precision@k、MRR。但 Agent 场景更重要的是端到端任务效果，所以还要看任务成功率、用户修改率、冲突率、工具调用正确率，以及人工抽样评估。比如检索命中了某条记忆，但导致排错日程，那从 Agent 角度仍然是失败。

  

持续学习最怕越学越偏。比如用户某天临时说“今天想早上学习”，系统不能立刻把它当成长期偏好。我的做法是把记忆分成弱记忆和强记忆：一次性表达先作为低置信度、短期有效的记忆；多次重复或用户显式确认后，才升级为长期偏好。同时要有时间衰减、冲突检测、用户可编辑、审计日志和回滚机制。这样系统可以学习用户，但不会因为一次错误抽取就长期污染行为。

  

用“帮我安排一周学习计划”这个场景来说，系统会先识别这是学习规划 + 日程安排任务。然后从记忆库里按 user_id、study_schedule 领域、未过期、当前 Agent 可用范围过滤候选记忆，再向量召回与学习计划相关的偏好，比如“晚饭后学习效率高”“Java 面试是近期目标”。同时从日程库结构化查询已有安排，比如健身、课程、会议。检索结果经过重排后，Maxwell 或学习规划 Agent 使用这些约束生成计划：优先晚饭后安排 Java 复习，但避开已有日程。如果召回到过期偏好或与用户当前指令冲突，就不采用。最后通过待确认卡片让用户确认，用户的修改再反过来更新偏好。这样 RAG 不是单纯把历史塞进 prompt，而是一个受边界控制、可评估、可修正的记忆检索系统。

  

### 参考回答要点

  

- 文档 RAG 和个人记忆 RAG 的差异：个人记忆更重视用户隔离、时效性、冲突、可解释和可删除。

- 推荐结构：结构化过滤 + 向量检索 + rerank。

- metadata 必须丰富，不能只有 text + embedding。

- 评估要看检索指标和端到端任务指标。

- 持续学习要防漂移：弱记忆、强记忆、确认、衰减、审计、回滚。

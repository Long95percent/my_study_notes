
面试官，关于 Tool Calling，我从三个层面说明：

**一、本质定义**

Tool Calling 是大模型在生成文本之外，输出**结构化调用请求**的能力。这个请求通常包含工具名称和参数 JSON。它不是模型天然具备的，而是通过**有监督微调（SFT）** 让模型学会在特定格式下输出。与 Function Calling 相比，Tool Calling 是更广义的抽象——不局限于“函数”，而是涵盖 API、脚本、工作流等任何可执行能力。

**二、工作流程**

标准流程是四步闭环：
1. **发起**：开发者将用户问题 + 可用工具的 JSON Schema（含名称、描述、参数结构）一起发给模型。
2. **决策**：模型判断需要调用工具时，不输出文本，而是返回一个结构化的调用对象（如 `{“tool”: “get_weather”, “args”: {“city”: “Beijing”}}`）。
3. **执行**：外部系统解析该对象，真实调用对应工具（HTTP、数据库等），拿到结果。
4. **回传**：将执行结果再发给模型，模型根据结果进行下一步操作或者生成最终的自然语言回复。

其中，通过 `tool_choice` 参数可以控制模式：`auto` 让模型自主决策；`required` 或指定工具名则强制模型必须调用工具（常用来做结构化数据提取）。主流模型（GPT-4o、Claude 3、Gemini 2.0）都支持**并行调用**，一次响应返回多个工具请求。

**三、落地的工程关键**

理解这个流程后，真正的挑战在于**如何稳定、安全地规模化落地**。这里有两个核心工程问题。

第一个是**生成稳定性**。模型输出的 JSON 很可能格式错误——缺括号、参数类型不对、甚至凭空捏造工具名。直接解析这种输出的失败率早期能到 15%~20%。业界成熟的解决方法是**语法约束解码**：在模型生成每个 token 时，动态限制它只能输出符合预设 JSON 语法树或工具 Schema 的 token。这样能把解析错误率压到 1% 以下。GPT 的 JSON Mode、LLaMA.cpp 的 grammar sampling 都属于这类。

第二个随之而来的问题是**安全治理**。Tool Calling 一旦稳定，模型就能真正执行动作——查库、发邮件、改配置。风险也随之暴露：单个调用可能合法，但两个调用串起来可能形成越权链条（例如先查普通数据，再调用管理接口）。因此架构上必须在**执行前**加一层拦截，而不是事后审计。常见做法是引入预执行防火墙，检查当前 Agent 或用户是否有权调用该工具，甚至检测工具调用序列是否匹配预设的安全策略。

此外，行业正在形成标准化协议来简化集成，比如 **Model Context Protocol (MCP)**，它定义了工具发现（`tools/list`）、调用（`tools/call`）和变更通知的标准消息格式。有了它，每个模型不需要分别适配不同的工具接口，工具也可以动态注册和更新。

**四、定位总结**

Tool Calling 是 Agent 系统的基础协议层，它将“模型决策”与“动作执行”解耦，使模型可以安全、可观测地调用外部能力，是实现可执行 Agent 的必要条件。

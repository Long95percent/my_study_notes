面试官，关于 whitelist，我从本质定义、核心职责、架构位置、关键工程机制以及与整个调用链路的关系五个层面来说明。

**一、本质定义**

whitelist 是一个**声明式的权限控制矩阵**，用于定义“哪个 Agent 可以调用哪个工具（以及在什么条件下）”。它是最小权限原则的直接体现——默认拒绝所有调用，只显式放行被允许的组合。与常见的 RBAC（基于角色的访问控制）不同，whitelist 更适用于 Agent 数量固定、职责边界清晰的场景，因为它直接枚举允许的（Agent, Tool）对，而不是通过角色间接推导。

**二、核心职责**

whitelist 承担三方面职责。第一，**准入判断**——在工具调用进入执行层之前，拦截请求并检查当前 Agent 是否被授权调用目标工具。第二，**条件约束**——除了简单的二元允许/拒绝，whitelist 还可以附加条件，比如仅允许在特定意图下调用、仅允许读取操作而不允许写入、或者对参数值进行约束（例如客服 Agent 只能查询订单状态，不能修改订单价格）。第三，**审计与告警**——对被拒绝的调用进行记录，便于发现越权尝试或配置错误。

**三、架构位置**

在调用链路中，whitelist 通常位于 **ToolRegistry 校验之后、tool_runtime 执行之前**。具体位置是：模型输出 Tool Calling → 解析请求 → ToolRegistry 校验参数格式和工具存在性 → **whitelist 鉴权** → 通过后进入 tool_runtime 执行。把 whitelist 放在这里有两个好处：一是在进入昂贵的执行沙箱之前就拦截掉非法请求，节省资源；二是鉴权逻辑与执行逻辑完全解耦，修改权限不需要改动 tool_runtime 的任何代码。

**四、关键工程机制**

这里有几个值得展开的点。

第一，**数据结构的选型**。最简单的 whitelist 是一个嵌套映射，例如 `Map[AgentID, Set[ToolName]]`。当需要支持条件约束时，可以升级为 `Map[AgentID, Map[ToolName, Policy]]`，其中 Policy 是一个包含“允许的操作类型”、“参数约束”、“可调用的时间段”等字段的结构。查询复杂度通常是 O(1) 或 O(log N)，对热路径影响极小。

第二，**动态更新与热加载**。Agent 的工具权限可能需要动态调整，而不重启服务。因此 whitelist 的实现通常会将其存储在配置中心或 Redis 中，并支持监听变更事件，实时更新内存中的权限表。同时需要版本控制或校验机制，防止并发更新导致的不一致。

第三，**与 intent_router 的联动**。纯粹基于 Agent 身份的 whitelist 可能过于粗粒度。更精细的设计是 whitelist 同时考虑当前意图。例如，同一个客服 Agent，在“查询订单”意图下可以调用 `query_order` 工具，在“修改订单”意图下则不被允许。这种设计需要 whitelist 在判断时能够接收“当前意图”作为额外上下文，形成 `(Agent, Intent, Tool)` 的三元组鉴权。

第四，**默认拒绝与显式放行**。whitelist 必须遵循默认拒绝原则——不在列表中的组合一律拒绝。这要求注册新工具或新 Agent 时必须主动配置 whitelist，否则调用会失败。实践中常配合自动化测试，检查是否有 Agent 声称支持某工具但未在 whitelist 中配置，避免运行时才发现问题。

**五、与整个调用链路的关系**

将前面讲过的组件串联起来：Tool Calling 定义了调用协议；ToolRegistry 负责工具注册、参数校验和元数据管理；whitelist 在参数合法之后、实际执行之前做了一次**安全关卡**；通过后再由 tool_runtime 负责执行隔离和策略保障。可以把 whitelist 理解为“权限层”，它的存在使得 Agent 即使能够输出某个工具的调用请求，也无法越权执行未授权的操作——这是多智能体系统安全架构的核心防线。

**总结**

whitelist 的本质是一个**轻量级的、面向 Agent 的访问控制列表**。它解决的核心问题是：在多智能体共享同一套工具池的情况下，如何确保每个 Agent 只能在其职责范围内行动。它的设计原则是简单、快速、可动态更新，并且默认拒绝一切未显式允许的调用。

## 一、FAISS (Facebook AI Similarity Search)

### 1. 一句话定位
> **FAISS 是一个高效的相似性搜索库**，由 Meta FAIR 团队开源，专注在单机或单卡上对海量向量进行极速的最近邻检索。

### 2. 核心原理（面试官可能会深入追问）
- **向量量化技术**：核心是 IVF (Inverted File Index) 和 PQ (Product Quantization)。
  - **IVF**：先通过 K-means 聚类将向量空间划分为多个聚类（Voronoi 单元），搜索时只探针最近的几个聚类，避免全量扫描。
  - **PQ**：将高维向量切分成多个子向量，每个子向量独立量化（用短码代替浮点数），压缩内存占用（比如从 4 bytes 压缩到 1 byte），同时利用预计算的距离表加速检索。
  - **HNSW** (可选)：基于图结构的索引，查询精度高，但内存消耗更大。
- **硬件加速**：支持 GPU（CUDA），利用并行计算加速距离计算和 k-selection。
- **距离度量**：支持 L2、内积（IP）等。

### 3. 优缺点
| 优点 | 缺点 |
|------|------|
| 极致的检索速度（内存/显存内计算） | 只是一个**库**，不是数据库：不支持 CRUD、持久化、分布式、高可用 |
| 内存占用非常灵活（通过 PQ 可以做到 10 倍压缩） | 数据量大时跨多机分布需要自己做 sharding |
| 支持 C++/Python/GPU，易集成 | 向量更新、删除很麻烦（重建索引或特殊处理） |
| 文档丰富，社区成熟 | 缺乏标量过滤、事务等数据库特性 |

### 4. 适用场景（面试中的“何时用它”）
- **学术实验 / 快速原型**：快速验证向量检索算法。
- **单机千万级内的向量检索**：比如 Embedding 数量在几百万内，机器内存足够。
- **作为本地 cache / embedding store**：配合 LangChain 等框架使用 `FAISS.from_documents`。
- **对纯向量距离计算性能要求极高，不需要数据管理**。

### 5. 技术深度点（加分项）
- **索引类型选择**：`IndexFlatL2`（暴力搜索，精度 100%） vs `IndexIVFPQ`（速度+压缩） vs `IndexHNSW`（高召回但内存大）。
- **GPU 使用注意事项**：显存不足时数据需分批；k-selection 在 GPU 上容易成为瓶颈，可调参数 `reserveVecs`。
- **量化误差**：PQ 压缩后精度会有损失，可通过粗量化+细量化平衡。

## 二、Milvus (Zilliz 开源的向量数据库)

### 1. 一句话定位
> **Milvus 是一个云原生的分布式向量数据库**，专为大规模生产环境设计，提供了完整的数据库能力（CRUD、标量过滤、高可用、水平扩展）。

### 2. 核心原理（面试重点）
- **存储计算分离**：数据持久化在对象存储（MinIO/S3）中，计算节点无状态，易于弹性伸缩。
- **索引架构**：支持多种索引类型（IVF_FLAT, HNSW, DiskANN, GPU 索引等），查询时可动态选择。
- **标量 - 向量混合搜索**：同时过滤标量字段（如 `tags = 'urgent'`）和向量相似度，通过 `search` 请求的 `expr` 参数实现，底层利用倒排或位图加速。
- **分段与流式处理**：数据按 Segment 管理，插入时不阻塞查询，后台自动合并。
- **一致性模型**：支持默认的 **强一致性**（读已写）和 **最终一致性**，可 trade-off。

### 3. 优缺点
| 优点 | 缺点 |
|------|------|
| 支持十亿/百亿级向量，水平扩展方便 | 部署运维较重（依赖 etcd, MinIO, Pulsar 等组件），有学习成本 |
| 提供完整的数据库功能（DDL/DML/权限/备份/监控） | 单机性能可能不如 FAISS（因为网络/持久化开销） |
| 混合搜索是原生支持，无需额外工程 | 小数据量（<100w）显得“重” |
| 云原生，K8s 友好，提供 Helm Chart | 社区版缺少某些企业特性（如动态 schema 管理） |

### 4. 适用场景（面试中强调企业级）
- **RAG 应用的知识库**：存放文档向量 + 元数据（时间、来源、标签）。
- **推荐系统**：用户/商品向量存储，结合标量过滤（如地区、价格区间）。
- **海量 AI Agent 长期记忆**：跨会话的用户画像、历史事件向量检索。
- **多租户需求**：用 `partition_key` 或 collection per tenant 实现隔离。

### 5. 技术深度点（加分项）
- **索引调优**：`nlist` (IVF 聚类数) 和 `nprobe` (搜索时探针数) 的平衡——增大 `nlist` 提升召回上限但增加索引时间，增大 `nprobe` 提升召回但降低速度。
- **DiskANN 索引**：Milvus 支持基于 SSD 的索引，内存不够时依然可以快速检索。
- **AutoIndex**：Milvus 2.3+ 可根据数据量自动选择最合适的索引，降低调优门槛。
- **Time Travel**：基于消息流（Pulsar）实现数据版本和时间点查询。

## 三、FAISS vs Milvus 对比（面试中总结）

| 维度 | FAISS | Milvus |
|------|-------|--------|
| **类型** | 相似性搜索库 | 分布式向量数据库 |
| **数据量级** | 适合 ≤ 1000w（单机） | 支持百亿级 |
| **数据操作** | 增删改需自己处理 | 原生 CRUD + upsert |
| **标量过滤** | 不支持（需外部做后过滤） | 原生支持混合搜索 |
| **分布式** | 无，需用户实现 | 原生支持，存储计算分离 |
| **持久化** | 无（需要用户序列化到磁盘） | 自动持久化到对象存储 |
| **学习门槛** | 低 | 中高 |
| **典型场景** | 研究、原型、单机高性能检索 | 生产级 RAG、推荐、大规模 Agent 记忆 |

### 面试金句（收尾）
> “如果我是做学术验证或单机原型，我会选择 FAISS；但在生产环境中，尤其是数据规模预计超过千万、需要混合查询和运维管理的场景，我会优先考虑 Milvus，必要时可使用全托管的 Zilliz Cloud 降低运维成本。”

## 四、可能的追问 & 如何回答

**Q1：FAISS 的 PQ 是如何减少内存的？**
> 答：将 128 维浮点向量切分为 16 个 8 维子向量，每个子向量用 256 个中心点（1 byte 索引）表示，原来一个浮点数 4 bytes，128 维就是 512 bytes；压缩后只需 16×1=16 bytes，压缩 32 倍。查询时使用预计算的查表法快速计算距离。

**Q2：Milvus 如何保证高可用？**
> 答：Milvus 的每个组件（coordinator, datanode, indexnode, querynode）都可多副本部署，依赖 etcd 做 leader 选举，查询节点有负载均衡；数据通过多副本或多租户配置，故障自动 failover。

**Q3：如果向量量很小（比如 10 万条），用谁？**
> 答：FAISS 更轻便，直接内存检索毫秒级；Milvus 需要维护独立集群，代价过高。建议 FAISS 或者简单的 `sklearn.neighbors` 都行。

**Q4：它们的替代品有哪些？**
> 答：还有 Qdrant（轻量级 Rust 向量数据库）、Weaviate、Pinecone（云服务）、pgvector（PostgreSQL 插件）。如果面试官问选型依据，可以说“根据数据规模、运维预算、是否需混合查询、以及团队熟悉的技术栈决定”。
